## 1. 背景与目标

### 1.1 背景

网易云音乐官方客户端/网页端缺少高效的「批量把歌曲加入指定歌单」能力：用户只能逐首搜索、逐首收藏，整理一个几十上百首的歌单非常耗时。常见场景包括：从外部清单（KTV 曲库、Excel 歌单、好友推荐列表）导入、把零散收藏归整到一个主题歌单等。

### 1.2 产品目标

做一个工具，让用户能够**一次性把一批歌曲准确地添加到自己的某个歌单中**。核心价值：

- **批量**：一次提交几十到几百首。
- **准确**：关键词搜索后能消歧（同名歌曲、翻唱、伴奏区分），避免加错版本。
- **可控**：添加前可预览匹配结果、人工修正；添加后给出清晰的成功/失败/未匹配报告。

### 1.3 非目标（本期不做）

- 不做音乐播放、下载、音质解析。
- 不做歌单之间的同步/双向更新。
- 不做多账号管理（仅支持当前授权的单账号）。
- 不做对非本人歌单的写操作。

## 2. 目标用户与典型场景

### 2.1 目标用户

- 重度网易云用户，习惯自己整理主题歌单。
- 有外部歌曲清单需要导入的用户（如从其它平台迁移、KTV/演出曲单整理）。

### 2.2 典型场景

1. **文件导入**：用户手上有一个 `txt`/`csv`，每行一首「歌名 - 歌手」，希望全部加进「跑步歌单」。
2. **关键词批量搜索**：用户粘贴一批歌名（可不带歌手），工具搜索匹配后批量加入。
3. **匹配修正**：对于搜索出多个结果的歌曲，用户在预览界面手动选择正确版本，再确认添加。

## 3. 产品形态与技术方案

### 3.1 形态

**桌面软件**：用 Electron 打包，把「前端页面 + 本地后端」封装进一个安装包（macOS `.app` / Windows `.exe`）。用户双击即用，体验等同普通 App；软件内部用内置浏览器内核渲染前端，并随软件启动一个仅监听本机的后端进程。

#### 形态选型依据（为什么不是纯网页）

「输网址即用」的纯网页方案无法成立，因为浏览器内的网页受三重限制，且任何一条单独成立即被拦住：

1. **CORS 跨域**：浏览器同源策略会拦截网页直接请求网易云接口的响应；该限制只约束浏览器脚本，后端发请求不受限。
2. **密钥无法保护**：前端代码对用户完全可见，`appKey/appSecret` 写进前端等于公开泄露。
3. **请求需签名**：开放平台接口要求 AppKey 签名，裸请求会被判为非法，签名逻辑需配合密钥在后端运行。

因此必须有一个「非浏览器的程序」去发请求。桌面软件把这个后端**内嵌到用户本地**，相比公网部署的优势：

- **无服务器成本与风控压力**：每个用户用自己的账号、在自己机器上发请求，不汇聚到统一服务器。
- **密钥与凭证留在本地**，不上公网。
- **零跨域问题**：本地后端发请求不受同源策略限制。

> 分发代价（一次性）：需分别为 macOS / Windows 打包；macOS 需签名与公证，否则首次打开会出现安全拦截提示。
>
> 框架选型：默认 **Electron**（前端与 Node 后端可直接复用，生态成熟）；若后续在意安装包体积，可评估迁移到 Tauri（系统 WebView + Rust，包体更小）。

### 3.2 整体架构

```text
┌──────────── 桌面软件（单一安装包）────────────┐
│  前端 UI（内置 WebView 渲染）                  │
│   导入 / 匹配预览 / 逐项修正 / 进度 / 报告       │
│        │  本地 IPC / localhost 通信            │
│  内置后端进程（仅监听本机）                      │
│   授权 · 签名 · 限流 · 重试 · 去重 · 结果聚合     │
└───────────────────────────────────────────────┘
                     ▼
            网易云开放平台 API
```

- **前端**：页面与交互（建议 React/Vue，按团队习惯），运行在软件内置浏览器内核中。
- **后端**：随软件启动的本地进程，封装授权、对开放平台的请求签名、并发与限流控制、错误重试、结果聚合。`appKey/appSecret` 仅存在本地后端，不进前端、不上公网。
- **接口适配器层（架构决策）**：后端将"对网易云的对接"抽象为统一接口适配器，对上层暴露稳定的能力契约（搜歌、搜专辑、取专辑曲目、取我的歌单、写入歌单），底层可挂接方案 A（官方开放平台）或方案 B（社区接口）两种实现。应用层（搜索匹配、批量、去重、限流、重试、报告）**不感知底层实现**，A/B 可通过配置切换，详见 3.3。
- **通信**：前后端通过本地 IPC 或 `localhost` 回环通信，端口不对外暴露。

### 3.3 与网易云的对接（接口调研结论）

> 调研时间：2026-06。官方文档正文需登录查看，以下结论综合官方平台公开信息与社区资料，**带 ⚠️ 项需在 M0 阶段实测确认**。

本项目所需的接口能力分四类：搜索歌曲、搜索专辑 + 获取专辑曲目、获取我的歌单、**向歌单写入歌曲**（核心）。经调研，存在两条可选对接路径：

#### 方案 A：官方开放平台（developer.music.163.com）

- **接入流程**：企业入驻 → 创建应用 → 审核 → 应用上线，**面向企业 / B 端**（车机、音箱等授权场景），个人开发者通常无法通过资质审核。
- **鉴权**：RSA 2048 / PKCS8 密钥对**签名** + `access_token`（非简单 Cookie），密钥在控制台生成。
- **能力匹配**：搜索、专辑、歌曲信息类接口具备；**⚠️ 但「操作用户个人歌单 / 向歌单写入歌曲」是否对第三方开放，未能确认**——这是决定方案 A 能否成立的关键，也是本项目最大不确定性。
- **优点**：官方授权、稳定合规。**缺点**：接入门槛高（需企业资质）、写歌单能力存疑。

#### 方案 B：社区开源接口（NeteaseCloudMusicApi 等）

- **原理**：通过 Cookie 登录态 + 伪造请求头调用网易云 Web/APP 接口（CSRF 方式），非官方授权。
- **能力匹配**（社区资料已验证存在对应接口）：
  - 搜索：`/search`（按关键词搜歌/专辑，`type` 区分）
  - 专辑曲目：`/album?id=`（获取整张专辑全部曲目）
  - 我的歌单：`/user/playlist`
  - 新建歌单：`/playlist/create`
  - **向歌单增删歌曲：`/playlist/tracks`，`op=add`，`pid`=歌单 ID，`tracks`=歌曲 ID（多个逗号分隔）** ← 满足核心写需求
- **优点**：能力完整、个人可用、本地 Cookie 登录即可。**缺点**：⚠️ 非官方授权、违反官方条款风险、接口随官方更新可能失效、有风控/封号风险。

#### 选型与架构决策

- **架构已定：接口适配器封装，A/B 可切换。** 后端定义统一的对接契约（搜歌 / 搜专辑 / 取专辑曲目 / 取我的歌单 / 写入歌单），方案 A 与方案 B 各实现一个适配器；应用层只依赖契约、不感知底层实现，可通过配置切换具体实现（见 3.2）。这样无论 M0 验证结果倾向哪一方案，上层代码都无需改动。
- 用户已选定"官方开放平台 API"为方向，故**优先验证并默认启用方案 A**，M0 第一要务是确认「写歌单」能力与企业接入资质是否可达。
- 若方案 A 的写能力不可用或资质不可达，**切换到方案 B 适配器作为回退**（功能可行，但需用户知悉并接受合规与稳定性风险）。
- 适配器需统一抽象差异点：鉴权方式（签名 vs Cookie）、返回字段映射（归一化为内部歌曲/专辑/歌单模型）、错误码与限流策略。

#### ⚠️ M0 待实测确认项

1. 方案 A 是否对第三方开放「向用户歌单写入歌曲」，单次歌曲数量上限与参数格式。
2. 方案 A 的企业入驻资质要求是否可满足；应用审核 / 上线周期。
3. 鉴权细节（RSA 签名流程、`access_token` 有效期与刷新）。
4. 接口频率限制（QPS / 日配额），用于设计限流策略。
5. 搜索接口返回字段，确认可用于消歧的维度（歌手、专辑、时长）。
6. 若回退方案 B：评估 Cookie 登录的获取方式、稳定性与合规边界，并与用户确认是否接受。

## 4. 功能需求

### 4.1 功能总览

| 编号 | 模块 | 优先级 |
|---|---|---|
| F1 | 授权登录 | P0 |
| F2 | 歌曲来源录入（文件导入 / 关键词列表） | P0 |
| F3 | 搜索与匹配 | P0 |
| F4 | 匹配结果预览与人工修正 | P0 |
| F5 | 选择目标歌单（点击导入后） | P0 |
| F6 | 批量添加执行（去重/限流/重试） | P0 |
| F7 | 结果报告与导出 | P1 |
| F8 | 历史任务记录 | P2 |

### 4.2 F1 授权登录

- 引导用户完成开放平台授权，后端保存访问凭证。
- 显示当前登录用户昵称，确认身份正确。
- 凭证失效时给出明确提示并支持重新授权。

### 4.3 F2 歌曲来源录入

录入前先选择**录入模式**，再录入条目；同一批为单一类型，避免类型歧义：

- **单曲模式**：每个条目是一首歌。
- **专辑模式**：每个条目是一张专辑，导入时整张展开为多首歌。

> 两种模式均支持批量；若既要加单曲又要加整张专辑，分两次录入即可（结果会合并进同一目标歌单的待处理列表）。

选定模式后，支持两种录入方式：

1. **文件导入**：上传 `txt` / `csv`，每行/每条为一个条目。
   - 单曲模式 `txt`：每行 `歌名 - 歌手`（仅 `歌名` 也可，匹配精度下降）。
   - 专辑模式 `txt`：每行 `专辑名 - 歌手`（歌手可选，用于提升匹配精度）。
   - `csv`：识别表头，单曲模式支持列 `title`（歌名，必填）、`artist`、`album`；专辑模式支持列 `album`（专辑名，必填）、`artist`。
   - 解析后展示「待处理列表」，可删除个别行。
2. **关键词列表**：文本框粘贴，每行一个条目，规则同对应模式的 `txt`。

#### 专辑模式的展开

- 对每个专辑条目，搜索专辑候选；存在多个同名专辑时，在预览区让用户选择正确的那一张（展示专辑封面、歌手、发行年份、曲目数辅助消歧）。
- 选定后调用「获取专辑曲目」接口，把整张专辑的歌曲并入待处理列表，每首带来源标记「来自专辑：XXX」。
- 专辑内曲目已携带准确歌曲 ID，直接标记为「已自动匹配」，无需逐首搜索。
- 支持一次录入多张专辑，展开后与列表中其他条目合并，统一进入后续匹配/添加流程。

录入约束：

- 单次任务建议上限（如 500 首）。**专辑按展开后的实际曲目数计入总量**，超出上限时给出提示。
- 自动去除空行、首尾空格、明显重复行；合并时按歌曲 ID 去重。

### 4.4 F3 搜索与匹配

- 对每首输入项调用搜索接口，取 Top-N 候选。
- **自动匹配规则**：
  - 同时提供歌名+歌手时，优先选歌名与歌手都高度吻合的候选，标记为「已自动匹配」。
  - 仅提供歌名、或存在多个相近候选时，标记为「需确认」。
  - 无任何候选时，标记为「未匹配」。
- 匹配状态分三类：`已自动匹配` / `需确认` / `未匹配`。

### 4.5 F4 匹配结果预览与人工修正

- 列表展示每首：输入文本、匹配状态、当前选中候选（歌名/歌手/专辑/时长）。
- 「需确认」项可展开候选下拉，手动选择正确版本。
- 「未匹配」项支持手动重新搜索（修改关键词再查）。
- 支持批量操作：全选「已自动匹配」项、剔除「未匹配」项。
- 显示统计：总数 / 已匹配 / 需确认 / 未匹配。
- **筛选与查找**：
  - 按匹配状态筛选：`全部` / `已自动匹配` / `需确认` / `未匹配`（可与统计数字联动，点击数字即筛选对应项）。
  - 按来源筛选：单曲条目 / 指定专辑（如「来自专辑：XXX」），便于在批量导入多专辑后定位某张专辑的曲目。
  - 关键词搜索：在当前列表内按歌名/歌手实时过滤。
  - 筛选后的批量操作仅作用于当前筛选结果（如「剔除当前所有未匹配项」「选中当前结果」）。
- 确认列表无误后，点击「导入」按钮，进入选择目标歌单（F5）。

### 4.6 F5 选择目标歌单

- 在用户点击「导入」按钮后触发，再选择把这批歌曲加入哪个歌单。
- 拉取当前用户「我创建的歌单」列表（仅展示可写歌单）。
- 展示歌单名、封面、当前歌曲数。
- 支持搜索/筛选歌单；支持「新建歌单」作为目标（P1）。
- 选定目标歌单并二次确认后，进入批量添加执行（F6）。

### 4.7 F6 批量添加执行

- 用户在 F5 选定目标歌单并确认后，开始执行批量添加任务。
- 仅对「已确认匹配」的歌曲执行添加。
- **去重**：添加前查询目标歌单已有歌曲，跳过已存在的（避免重复，并在报告中标记「已存在」）。
- **分批**：按接口单次上限分批提交。
- **限流**：按接口频率限制控制请求节奏，避免触发风控。
- **重试**：失败项自动重试（指数退避，上限可配）。
- **进度**：实时展示进度条与「成功/失败/跳过」实时计数，可中途取消。

### 4.8 F7 结果报告与导出

- 任务结束展示汇总：成功数、失败数、跳过（已存在）数、未匹配数。
- 明细可按状态筛选；失败项显示原因。
- 支持把「失败 + 未匹配」清单导出为 `csv`，便于用户后续处理或重试。

### 4.9 F8 历史任务记录（P2）

- 本地保存近期任务的概要与报告，支持查看与对未完成项重试。

## 5. 页面与交互结构

整体采用「**单主页面 + 模态弹窗**」结构：主页面承载核心的搜索与匹配列表，所有阶段性、一次性的操作以弹窗形式叠加，避免页面跳转、保持上下文。

### 5.1 主页面：搜索与匹配列表

应用唯一的常驻页面，是用户停留时间最长的工作区。

- **顶部栏**：应用名 / 当前登录账号与头像（未登录显示「登录」入口）/「录入歌曲」按钮 /「导入到歌单」主操作按钮。
- **统计与筛选区**：总数 / 已匹配 / 需确认 / 未匹配 统计（点击即筛选）；状态筛选、来源筛选（单曲 / 指定专辑）、列表内关键词查找。
- **主体：匹配列表**：每行展示输入文本、匹配状态、当前选中候选（歌名 / 歌手 / 专辑 / 时长）；行内可展开候选下拉修正、对未匹配项重新搜索；支持多选与批量操作（仅作用于当前筛选结果）。
- **空状态**：未录入任何歌曲时，引导用户点击「录入歌曲」。

### 5.2 弹窗（4 个）

| 弹窗 | 触发时机 | 主要内容 |
|---|---|---|
| **① 授权登录** | 应用启动未登录时自动弹出；或点顶部「登录」 | 网易云授权登录入口、登录状态与结果提示 |
| **② 录入歌曲** | 点顶部「录入歌曲」 | 先选录入模式（单曲 / 专辑），再选来源（文件导入 / 关键词列表），录入后并入主页面列表 |
| **③ 选择目标歌单** | 点「导入到歌单」主按钮后 | 我创建的可写歌单列表（名称/封面/歌曲数）、搜索筛选、新建歌单、二次确认 |
| **④ 导入结果** | 批量添加执行完成后 | 进度（执行中）→ 结果汇总（成功/失败/跳过）、失败与未匹配清单、导出入口 |

交互约定：

- 同一时刻仅展示一个弹窗；弹窗打开时主页面置灰、不可操作。
- ②③弹窗可取消返回主页面且不丢失已录入/已修正的列表数据；④弹窗在执行中不可随意关闭（防止中断写入），完成后可关闭。
- 授权弹窗①是其余操作的前置：未登录时点击录入/导入应先唤起①。

## 6. 关键流程

### 6.1 主流程

```text
授权登录
  → 选择录入模式（单曲 / 专辑）并录入歌曲来源（文件 / 粘贴）
  → 搜索匹配
  → 预览、筛选并人工修正
  → 点击「导入」按钮
  → 选择目标歌单（可新建）并二次确认
  → 执行批量添加（去重 → 分批 → 限流 → 重试）
  → 查看结果报告（可导出失败/未匹配清单）
```

### 6.2 异常与边界

| 场景 | 处理 |
|---|---|
| 授权失效 | 暂停任务，提示重新授权，已完成进度不丢失 |
| 接口限流/风控 | 自动降速重试；连续失败则暂停并提示 |
| 歌曲无搜索结果 | 标记「未匹配」，不阻断其他歌曲 |
| 歌曲已在歌单中 | 跳过并在报告标记「已存在」 |
| 单次数量超上限 | 录入阶段提示，引导拆分任务 |
| 网络中断 | 当前批失败可重试，已成功部分不重复添加 |

## 7. 非功能需求

- **安全**：`appKey/appSecret` 与用户凭证仅存后端，不下发前端、不写日志；本地存储凭证需加密。
- **合规**：仅对用户本人歌单做写操作；遵守开放平台调用频率与使用条款；不绕过官方风控机制。
- **性能**：500 首规模任务应能在接口限流允许范围内稳定完成；前端长列表需虚拟滚动。
- **可用性**：核心流程对非技术用户友好；关键操作（执行添加）有二次确认。
- **可观测**：后端记录任务级日志（不含敏感凭证），便于排查失败原因。
